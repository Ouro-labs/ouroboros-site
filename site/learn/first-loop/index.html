3. 아이디어 정리와 Interview

만들고 싶은 것을 한 줄로 말하는 데서 시작합니다. 잘 다듬은 프롬프트를 준비할 필요는 없습니다. 부족한 부분을 찾는 일이 Interview의 역할입니다.

Ouroboros 0.50.4upstream 456a134Ubuntu 24.04 · Claude Code 2.1.2102026-07-16 확인

3.1 tutorial로 흐름 확인

ooo tutorial

tutorial은 아이디어를 입력하고 몇 가지 질문에 답한 뒤 간단한 Seed를 확인하는 실습입니다. 실제 작업 전에 Interview의 질문 방식과 출력 형식을 확인할 때 사용합니다.

3.2 예제 아이디어로 Interview 시작

ooo interview "오늘 할 일을 추가하고 완료 표시할 수 있는 간단한 웹앱을 만들고 싶어요."

Interview는 결과에 영향을 주는 결정부터 확인합니다. 아래는 위 명령을 실행했을 때의 흐름을 설명하기 위한 가정 예시입니다. 실제 질문의 수와 순서는 아이디어에 따라 달라집니다.

Ouroboros“브라우저를 닫았다가 다시 열어도 할 일 목록이 남아 있어야 하나요?”

사용자“네, 새로고침해도 남아야 해요.”

Ambiguity Score 0.52 → 0.38 — 저장 방식 결정이 고정된 상태.

Ouroboros“완료한 일은 목록에서 지우나요, 완료 표시만 바꾸나요?”

사용자“지우지 말고 줄을 그어서 남겨 주세요.”

Ambiguity Score 0.38 → 0.24 — 완료 처리 방식이 고정된 상태.

Ouroboros“여러 사람이 함께 쓰나요, 한 기기에서 한 사람만 쓰나요?”

사용자“한 사람만 써요. 계정은 필요 없어요.”

Ambiguity Score 0.24 → 0.15 — 0.2 이하로 내려와 Seed 생성 준비 완료.

각 답이 선택 하나를 고정합니다. 목표, 제약, 성공 기준이 또렷해질수록 Ambiguity Score가 내려가고, 0.2 이하가 되면 Seed를 만들 준비가 된 것입니다.

질문 수보다 답의 정확성이 중요합니다.
처음 요청에 없던 결정을 코딩 전에 확인하면 수정해야 할 범위가 줄어듭니다.

3.3 Seed 생성과 저장 확인

Interview가 끝나면 출력에 남은 session_id를 넣어 Seed를 생성하거나 이미 만든 Seed를 조회합니다.

ooo seed [session_id]

생성된 Seed는 ~/.ouroboros/ 아래에 YAML 파일로 저장됩니다. 실행을 맡기기 전에 이 파일을 직접 열어, 4장의 기준으로 Goal, Constraints, Acceptance Criteria가 자신의 답과 일치하는지 확인합니다.

3.4 수동 실행과 자동 실행 선택

질문마다 직접 답하려면 interview를 사용합니다. 질문과 Seed 작성을 자동으로 진행하고 바로 실행까지 맡기려면 auto를 사용합니다.

ooo auto "오늘 할 일을 추가하고 완료 표시할 수 있는 간단한 웹앱을 만들고 싶어요."

auto도 아무 Seed나 실행하지 않습니다. Seed가 A등급을 통과해야 실행을 시작하고, 중간에 멈추면 auto_session_id를 남깁니다. 중단된 자동 세션은 아래 명령으로 재개합니다.

ooo auto --resume <auto_session_id>

설계만 먼저 보고 싶다면 실행을 건너뜁니다.

ooo auto "오늘 할 일을 추가하고 완료 표시할 수 있는 간단한 웹앱을 만들고 싶어요." --skip-run

이 문서의 검증 범위는 플러그인 설치까지 실측이며, 이 장의 문답과 점수는 공식 문서 동작에 맞춘 가정 예시입니다. 전체 고지는 가이드 홈에 있습니다.

04 · SEED

4. Seed 확인

Seed는 Interview에서 합의한 내용을 실행 전에 고정한 명세입니다. 코드보다 먼저 읽고, 생각과 다르면 실행 전에 고칩니다.

아래 YAML은 3장의 문답을 가정해 공식 Seed 필드에 맞춘 가이드용 예시입니다. 인증된 세션에서 실제 생성한 결과가 아닙니다. ooo seed가 만든 실제 파일은 ~/.ouroboros/에서 같은 구조로 확인할 수 있습니다.

goal: 오늘 할 일을 추가하고 완료 표시하는 웹앱 만들기
constraints:
  - 한 기기에서 한 사람만 사용
  - 브라우저 localStorage에 저장
  - 별도 서버와 계정 없이 작동
acceptance_criteria:
  - 빈 제목은 추가되지 않는다
  - 할 일을 추가하면 새로고침 뒤에도 남는다
  - 체크하면 완료 상태가 즉시 보이고 저장된다
ontology_schema:
  name: TodoList
  fields:
    - name: title
      type: string
    - name: completed
      type: boolean
evaluation_principles:
  - name: completeness
    description: 모든 완료 기준이 실제로 작동한다
    weight: 1.0
exit_conditions:
  - name: all_criteria_met
    description: 모든 완료 기준 통과
    criteria: 100% of acceptance criteria are satisfied
metadata:
  ambiguity_score: 0.15

4.1 Goal: 만들 결과

“예쁜 앱” 대신 사용자가 무엇을 할 수 있어야 하는지 적습니다. 이번 Goal에는 추가와 완료 표시가 있고, 로그인이나 협업은 없습니다.

4.2 Constraints와 Non-goals: 작업 범위

Constraints는 반드시 지킬 제한입니다. 한 사람, 한 기기, localStorage라는 답이 기술 선택 범위를 줄입니다. Non-goals는 만들지 않기로 한 일입니다. 이번에는 계정, 여러 기기 동기화, 마감일, 삭제를 넣지 않습니다.

Ledger는 결정의 근거를 저장합니다.
독립 터미널에서 auto --show-ledger를 실행하면 질문 과정에서 확인한 가정과 제외 범위를 볼 수 있습니다. “단일 사용자”가 사용자의 답인지 AI의 추측인지 확인할 때 사용합니다.

4.3 Acceptance Criteria: 완료 기준

“쓰기 편하다”는 통과 여부를 판정하기 어렵습니다. 빈 제목 차단, 새로고침 뒤 유지, 체크 상태 저장처럼 화면에서 직접 확인할 수 있게 씁니다. 각 항목은 5장에서 결과물을 열어 하나씩 검증하는 기준이 됩니다.

4.4 Ambiguity Score: 미결정 항목

목표, 제약, 성공 기준의 명확도를 합쳐 계산합니다. 0에 가까울수록 분명하며, 0.2 이하가 자동 실행을 허용하는 관문입니다. 점수가 낮아도 내용이 틀리면 실행해서는 안 되므로 숫자와 문장을 함께 봅니다.

4.5 Seed 수정

예를 들어 “완료한 일은 목록에 남긴다”라고 답했는데 Seed에 자동 삭제로 적혀 있다면 수정해야 합니다. 제안된 변경은 사용자가 승인한 뒤 YAML에 반영됩니다. 수정된 Seed와 점수를 다시 확인한 뒤 실행합니다.
