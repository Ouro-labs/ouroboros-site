5. 실행과 평가

Seed를 승인했다면 runtime에 실행을 맡깁니다. 실행 성공 문구로 끝내지 않고, 화면과 테스트를 실제로 확인해 평가까지 통과시킵니다.

Ouroboros 0.50.4upstream 456a134Ubuntu 24.04 · Claude Code 2.1.2102026-07-16 확인

5.1 run으로 실행 시작

ooo run

최근 Seed나 전달한 YAML을 검증하고 배경 실행을 시작합니다. 실제 파일 작성과 명령 실행은 설정한 runtime, 여기서는 Claude Code가 맡습니다.

3장에서 ooo auto를 실행했다면 A등급 Seed가 만들어진 뒤 실행이 이어집니다. --skip-run으로 멈췄다면 Seed를 확인한 뒤 ooo run을 입력합니다.

5.2 실행 ID 확인

명령마다 받는 ID가 다르므로, 출력에 나온 값을 역할별로 구분해 복사해 둡니다.

ID출처사용처
job_idrun 시작 출력배경 작업의 상태·결과 조회
session_idInterview·run 출력ooo status, ooo evaluate, 세션 재개
execution_idrun 시작 출력ooo cancel, ouroboros cancel execution
lineage_idevolve 출력ooo evolve --status, ooo ralph --lineage-id

5.3 실행 단계 확인

Discover저장소와 환경, 요구의 빈칸을 확인합니다.
Define무엇을 풀고 무엇을 빼는지 좁힙니다.
Design구조와 구현 순서를 정합니다.
Deliver파일을 만들고 테스트와 실제 동작을 확인합니다.

네 단계는 진행 표시 이름이 아닙니다. 이해하지 않은 채 구현으로 뛰는 일을 막는 실행 순서입니다.

5.4 진행 상태와 변경 확인

ooo status <session_id>

현재 상태와 진행을 읽습니다. 원래 Seed에서 멀어졌는지 물으면 Goal 50%, Constraints 30%, Ontology 20% 가중으로 Drift를 잽니다. 0.3을 넘으면 경계를 다시 확인할 시점입니다.

Drift가 낮아도 기능이 맞다는 뜻은 아닙니다.
원래 방향을 지켰다는 신호입니다. 작동과 완료 기준은 평가에서 따로 확인합니다.

5.5 실행 결과 평가

ooo evaluate <session_id>
  1. Mechanical. lint, build, test를 실행합니다. 실패하면 다음 평가로 넘어가지 않습니다.
  2. Semantic. 결과가 Goal과 Acceptance Criteria를 충족했는지 확인합니다.
  3. Multi-Model Consensus. 추가 판단이 필요한 경우 여러 모델의 결과를 비교합니다.
Consensus가 실행되는 조건

Seed 수정, ontology 변경, Goal 재해석, Drift 0.3 초과, 불확실성 0.3 초과, 별도 제안 채택 중 하나에 해당하면 실행됩니다. Semantic 점수가 0.8 이상이고 해당 조건이 없으면 생략됩니다.

run이 성공해도 정식 평가 전 상태는 executed_unverified일 수 있습니다. 평가가 자동으로 이어지지 않았다면 위 명령을 직접 실행합니다.

5.6 결과물 직접 확인

평가 통과 문구만 읽고 끝내지 않습니다. 출력 확인, 판정 확인, 직접 검증의 순서로 마무리합니다.

  1. 출력 확인 — run 완료 메시지에서 결과물이 생성된 폴더 경로를 확인합니다. 이 예제라면 할 일 앱의 HTML 파일이 있는 작업 폴더입니다.
  2. 판정 확인 — ooo evaluate <session_id>의 세 단계 결과와 남은 지적을 읽습니다.
  3. 직접 검증 — 생성된 앱을 브라우저에서 직접 엽니다. 이 예제라면 작업 폴더의 index.html을 여는 것으로 충분합니다. 그 화면에서 Seed의 Acceptance Criteria를 하나씩 재현합니다. 빈 제목이 추가되지 않는지, 추가한 할 일이 새로고침 뒤에도 남는지, 체크한 완료 상태가 즉시 보이고 저장되는지 확인합니다.

5.7 qa로 단건 확인

ooo qa

qa는 최근 결과나 지정 파일을 품질 기준과 한 번 비교해 PASS, REVISE, FAIL을 돌려줍니다. 3단계 정식 실행 검증인 evaluate와 역할이 다릅니다. 화면이나 API처럼 실제 행동이 있는 결과는 소스만 읽지 말고 직접 실행한 증거를 함께 넣습니다.

이 문서의 검증 범위는 플러그인 설치까지 실측이며, 이 장의 실행·평가 절차는 공식 문서 기준으로 서술합니다. 전체 고지는 가이드 홈에 있습니다.

6. 수정과 프로젝트 이관

평가에서 찾은 문제를 다음 Seed에 반영합니다. 예제 작업을 마친 뒤에는 같은 절차를 기존 저장소에 적용할 수 있습니다.

6.1 evolve로 한 번 수정

ooo evolve "할 일 앱 평가 결과를 반영해 완성도를 높여줘"

Wonder는 평가에서 확인하지 못한 항목을 찾습니다. Reflect는 그 결과를 바탕으로 Seed의 목표, 제약, 완료 기준 변경안을 만듭니다. 연속된 Seed의 ontology 유사도가 0.95 이상이면 변화가 거의 없는 상태로 판단하며, 최대 30세대에서 멈춥니다.

코드를 다시 만들지 않고 Seed 변경안만 확인하려면 --no-execute를 붙입니다. 이어진 실행 목록은 아래 명령으로 확인합니다.

ooo evolve --status <lineage_id>

6.2 Ralph로 반복 실행

ooo ralph --lineage-id <lineage_id>

Ralph는 완료 기준 충족, 변화가 거의 없는 상태, 사용자 취소, 복구할 수 없는 실패, 반복 횟수 제한 중 하나에 도달할 때까지 배경에서 실행을 반복합니다. 먼저 Interview와 Seed를 만든 뒤 출력된 lineage_id를 사용합니다.

사람이 멈출 이유도 분명합니다.
완료 기준이 이미 충족됐거나, 같은 질문이 반복되거나, 비용에 비해 바뀌는 내용이 작거나, 제품 방향을 사람이 다시 정해야 한다면 중단합니다. 자동 반복 횟수가 품질을 보장하지 않습니다.

6.3 첫 작업 종료

아이디어가 Interview를 거쳐 Seed가 됐고, runtime이 만들고 평가했으며, 결과물을 직접 열어 완료 기준을 확인했습니다. 평가에서 배운 것은 evolve의 다음 입력이 됩니다. 여기까지가 첫 순환입니다.

6.4 기존 프로젝트에 적용

아래 명령들은 첫 예제를 닫은 뒤 자기 프로젝트와 팀 작업으로 넘어갈 때 사용합니다. 첫 순환을 완주하기 전에는 필요하지 않습니다.

6.4.1 brownfield: 기존 저장소 연결

ooo brownfield

기존 git 저장소와 worktree를 찾아 Interview의 기본 컨텍스트로 등록합니다. 새 프로젝트처럼 추측하지 않고 현재 코드, 제약, 빌드 명령을 바탕으로 질문하게 됩니다. 여러 저장소가 나오면 실제 작업할 저장소만 기본값으로 선택합니다.

6.4.2 pm: 제품 요구 정리

ooo pm

제품 요구사항을 질문하고 PRD와 작업 전달 자료를 만듭니다. 이 자료는 곧바로 실행할 Seed가 아닙니다. 출력된 경로를 ooo interview에 넣고 ooo seed로 실행 명세를 다시 만듭니다.

6.4.3 publish: 팀 작업 발행

ooo publish <seed_path>

Seed의 Goal을 Epic으로, Acceptance Criteria를 작업 이슈로 나눠 GitHub에 발행합니다. gh 설치와 로그인이 필요하고, 실제 생성 전에 대상 저장소와 이슈 구조를 확인합니다.

기존 프로젝트에서도 Interview, Seed, 실행, 평가, 수정 순서를 유지합니다. 기능 이름을 전부 외울 필요는 없습니다.
