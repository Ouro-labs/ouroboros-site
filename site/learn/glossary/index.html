12. 용어사전

제품 용어를 짧게 정의합니다. 각 항목 끝의 링크에서 실제 사용 절차를 확인합니다.

Ouroboros 0.50.4upstream 456a1342026-07-16 문서·코드 확인

12.1 명세

Interview

아이디어에서 아직 정해지지 않은 결정을 질문으로 확인하는 기능입니다. 5장

Ambiguity Score

요구사항에 남은 모호함을 0~1로 나타낸 값입니다. 0.2 이하면 Seed를 만들 수 있습니다. 5장

Seed

목표, 제약, 완료 기준, 데이터 구조를 담은 작업 기준 YAML 파일입니다. runtime과 평가가 모두 이 파일을 기준으로 동작합니다. 6장

Goal

이번 실행이 만들어야 할 결과를 적은 Seed 필드입니다. 6장

Constraints

실행이 반드시 지켜야 할 제한을 담은 Seed 필드입니다. 6장

Non-goals

이번에 만들지 않기로 한 일입니다. Seed 본문이 아니라 Ledger에 기록됩니다. 6장

Acceptance Criteria

작업 완료를 판정하는 기준 목록입니다. 화면이나 테스트에서 직접 확인 가능한 문장으로 씁니다. 6장

Ontology

작업이 다루는 데이터의 이름과 타입을 정한 구조입니다. 6장

A등급

차단 사유가 없고 품질 점수가 기준을 통과해 실행 가능한 Seed 상태입니다. ooo auto는 A등급에서만 실행을 시작합니다. 6장

Ledger

질문 과정에서 확인한 가정, 제외 범위, 결정 근거를 남긴 기록입니다. 6장

12.2 실행

Core

Interview, Seed, 실행 관리, 평가를 담당하는 Ouroboros 엔진입니다. 3장

runtime

실제 파일과 코드를 다루는 AI 코딩 도구입니다. Claude Code, Codex CLI 등이 있습니다. 3장

MCP

runtime이 Ouroboros Core의 기능을 호출하게 잇는 통신 규약입니다. 4장

Double Diamond

Discover, Define, Design, Deliver 순서로 진행하는 실행 단계입니다. 7장

12.3 평가

Mechanical

lint, build, test 같은 명령을 실행해 결과를 확인하는 평가 1단계입니다. 8장

Semantic

결과물이 Acceptance Criteria를 충족했는지 LLM이 판정하는 평가 2단계입니다. 8장

Consensus

여러 모델이 독립 투표로 승인 여부를 정하는 평가 3단계입니다. 정해진 조건에서만 실행됩니다. 8장

Drift

결과가 원래 Seed의 의도에서 멀어진 정도를 나타내는 0~1 값입니다. Goal 50%, Constraints 30%, Ontology 20% 가중으로 계산합니다. 8장

12.4 반복

Evolve

평가에서 확인한 문제를 Seed에 반영해 다시 실행하는 기능입니다. 9장

lineage

같은 작업에서 이어진 Seed 수정 실행들을 묶은 이력입니다. 9장

Convergence

반복 실행을 해도 Seed의 데이터 구조가 거의 바뀌지 않는 상태입니다. 연속 세대의 Ontology 유사도 0.95 이상이 기준입니다. 9장

Stagnation

반복해도 결과가 더 나아지지 않는 상태입니다. 반복을 멈추고 접근을 바꾸는 신호로 사용합니다. 9장

Ralph

종료 조건에 도달할 때까지 수정 실행을 배경에서 반복하는 기능입니다. 9장

12.5 기록과 ID

EventStore

모든 실행 이벤트를 저장하는 로컬 SQLite 데이터베이스입니다. 세션 복구에 사용합니다. 3장

session_id · job_id · execution_id · lineage_id · auto_session_id

각각 세션 조회·평가, 배경 작업 조회, 실행 취소, 수정 이력, auto 재개에 사용하는 ID입니다. 7장
